**Problem #1 Thread-Safe Counter Application**

Build a multithreaded counter application that demonstrates your understanding of std::thread, std::mutex, std::lock_guard, race conditions, and std::atomic. You will first observe a race condition, then fix it.

**Requirements**

**Part A: Demonstrate the Race Condition**

- Create a global integer counter initialized to 0.
- Write an increment() function that increments counter 100,000 times in a loop.
- Launch 4 threads, each running increment().
- After all threads finish, print the expected value (400,000) and the actual value.
- Run the program 5 times and record the actual values in a comment at the top of your file.

**Part B: Fix with std::mutex and std::lock_guard**

- Add a global std::mutex.
- Write an incrementSafe() function that uses std::lock_guard to protect the counter.
- Launch 4 threads running incrementSafe().
- Verify the result is always exactly 400,000.
- Measure the execution time using &lt;chrono&gt; and print it.

**Part C: Fix with std::atomic**

- Declare a std::atomic&lt;int&gt; atomicCounter.
- Write an incrementAtomic() function that increments it 100,000 times.
- Launch 4 threads and verify the result is always 400,000.
- Measure and print the execution time. Compare with Part B in a comment.

**Part D: Qt GUI with Progress**

- Create a Qt application with a QProgressBar and a "Start" button.
- Use the Worker/moveToThread pattern to run the counting in a background thread.
- The worker should emit a progressChanged signal every 10,000 increments.
- Connect the signal to the QProgressBar to show progress.
- The UI must remain responsive during counting (not frozen).

**Sample Output:**

Shows the Qt GUI at top with a progress bar at 72% (288,000 / 400,000 increments), the grayed-out "Start counting" button, and a console panel below with all three parts: Part A showing the race condition (expected 400,000 vs. actual 312,005), Part B showing the mutex fix (always 400,000, 45ms), and Part C showing the atomic fix (always 400,000, 12ms with the speed comparison).

**Problem #2 Producer-Consumer Task Processor**

Build a Producer-Consumer application using condition variables, a thread-safe queue, and the Qt Worker/moveToThread pattern. The application simulates a task processing pipeline where tasks are submitted to a queue and multiple workers consume and process them.

**Requirements**

**Part A: Thread-Safe Queue**

- Create a template class ThreadSafeQueue&lt;T&gt; that wraps std::queue&lt;T&gt;.
- Implement push(const T& item) that locks a mutex, pushes the item, and notifies a waiting consumer via condition_variable.
- Implement pop(T& item) that locks a mutex, waits on the condition_variable until data is available, then pops and returns the item.
- Implement a shutdown() method that notifies all waiting threads to exit.
- Use std::unique_lock and std::condition_variable correctly.

**Part B: Console Producer-Consumer**

- Create a producer function that pushes 20 numbered tasks (strings like "Task-1", "Task-2", etc.) into the queue, sleeping 100ms between each.
- Create a consumer function that pops tasks from the queue and prints them with the consumer ID.
- Run 1 producer thread and 3 consumer threads.
- After the producer finishes, call shutdown() to stop the consumers.
- Print a summary showing how many tasks each consumer processed.

**Part C: Qt GUI Task Processor**

- Create a Qt main window with: a QLineEdit for entering task names, an "Add Task" button, a QListWidget showing the task queue, a QListWidget showing completed tasks, a QLabel showing the number of active workers.
- Use the Worker/moveToThread pattern for 2 consumer workers.
- Each worker should emit signals: taskStarted(QString), taskCompleted(QString), to update the GUI.
- Simulate processing time (sleep 500ms-1500ms per task, randomized).
- When a task is completed, move it from the queue list to the completed list.
